# Audio Code

## Why We Built This

We believe coding help should meet people where real work happens: in the terminal, in the middle of a train of thought, and often before there is time to explain everything perfectly. Too many developer tools force people to translate messy intent into rigid steps, breaking momentum exactly when clarity matters most.

We built Audio Code to make software work feel more conversational without making it careless. The goal is not to remove human judgment. The goal is to shorten the distance between an idea, a question, and a concrete change, especially when speaking is faster or more natural than typing.

---

## How We Approach This

- **Human direction first** — The system should amplify a developer's intent, not replace it.
- **Action through real tools** — Help must result in observable work in the actual project, not empty advice detached from the code.
- **Safety before convenience** — Permissions, visibility, and clear boundaries matter more than silent automation.
- **Voice without losing precision** — Spoken input should be practical for coding work, but never at the cost of control or reviewability.
- **Collaboration over magic** — Sessions, teammates, and remote handoff should make work easier to coordinate, not harder to understand.

---

## What It Does

### Core Capabilities

- Provides a terminal-based coding agent that can inspect, explain, and change code through explicit tools.
- Adds voice input so a developer can speak intent when that is faster than typing.
- Supports remote sessions and teammate-style workflows so work can continue across contexts and collaborators.
- Helps manage configuration, plugins, project context, and task flow around the core coding loop.

### What This Is Not

This project does **not**:
- Aim to replace developer ownership or final judgment.
- Optimize for fully invisible automation that acts without clear boundaries.
- Serve as a generic chat product detached from real files, commands, and repositories.

---

## Who This Is For

- **Developers in the terminal** — People who want direct help inside their real working environment.
- **Voice-first or accessibility-minded users** — People who benefit when coding workflows can begin with speech instead of typing.
- **Teams coordinating AI-assisted work** — Groups that need sessions, handoffs, and parallel helpers without losing operational control.
